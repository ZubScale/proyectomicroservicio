01 — Introducción

Vista general del proyecto

Este proyecto implementa un sistema de e-commerce dividido en cuatro microservicios independientes que se comunican entre sí exclusivamente a través de HTTP.

Cada microservicio es una aplicación FastAPI completamente autónoma con su propia base de datos, su propio proceso y su propia responsabilidad de negocio. Ningún servicio accede directamente a la base de datos de otro — la única forma de interactuar es a través de sus APIs.

Un API Gateway actúa como punto de entrada único: el cliente siempre habla con el Gateway en el puerto 8000, y el Gateway enruta cada petición al servicio correcto.

Usuarios

Puerto 8001 · usuarios.db

Registro y consulta de usuarios. Gestiona nombres, emails y contraseñas.

Productos

Puerto 8002 · productos.db

Catálogo de productos con control de inventario (stock).

Pedidos

Puerto 8003 · pedidos.db

Orquesta la creación de pedidos llamando a Usuarios y Productos.

API Gateway

Puerto 8000 · sin base de datos

Punto de entrada único. Enruta peticiones y añade trazabilidad.

02 — Organización

Estructura del proyecto

Cada servicio es una carpeta independiente con sus propios archivos. Esta separación refleja el principio de responsabilidad única: cada carpeta contiene todo lo necesario para un dominio de negocio.

ecommerce/ ├── gateway/ │ ├── __init__.py │ └── main.py # Puerto 8000 — enruta al resto │ ├── usuarios/ │ ├── __init__.py │ ├── main.py # Puerto 8001 — endpoints REST │ ├── models.py # Tabla de la base de datos │ └── database.py # Conexión a SQLite │ ├── productos/ │ ├── __init__.py │ ├── main.py # Puerto 8002 — endpoints REST │ ├── models.py # Tabla de la base de datos │ └── database.py # Conexión a SQLite │ ├── pedidos/ │ ├── __init__.py │ ├── main.py # Puerto 8003 — orquesta otros servicios │ ├── models.py # Tabla de la base de datos │ ├── database.py # Conexión a SQLite │ └── schemas.py # Modelos Pydantic separados │ ├── demo.py # Demostración de los 4 servicios ├── reset.py # Limpia todas las bases de datos ├── requirements.txt # Dependencias del proyecto ├── Dockerfile # Imagen Docker └── docker-compose.yml # Levanta todos los servicios
¿Por qué __init__.py?

El archivo __init__.py (vacío) le indica a Python que la carpeta es un paquete. Esto permite usar imports relativos dentro de cada servicio, como from .database import Base, que es más robusto que imports absolutos.

03 — Diseño

Arquitectura del sistema

El cliente solo conoce el Gateway. El Gateway conoce las URLs de los servicios internos. Los servicios internos se llaman entre sí cuando la lógica de negocio lo requiere.

CLIENTE (browser / demo.py) | | HTTP :8000 ▼ ┌─────────────────────────┐ │ API GATEWAY │ ← enruta · trazabilidad │ :8000 │ └──────────┬──────────────┘ | ┌────────┼─────────┐ ▼ ▼ ▼ ┌─────────┐ ┌──────────┐ ┌─────────┐ │Usuarios │ │Productos │ │Pedidos │ │ :8001 │ │ :8002 │ │ :8003 │ └────┬────┘ └────┬─────┘ └────┬────┘ | | | | | llama a Usuarios | | llama a Productos ▼ ▼ ▼ usuarios.db productos.db pedidos.db (SQLite) (SQLite) (SQLite)

El servicio de Pedidos es el más interesante: cuando recibe un pedido, internamente llama al servicio de Usuarios para verificar que existe, y al de Productos para verificar stock y descontarlo. El cliente no sabe que esto ocurre.

Principio clave — Base de datos propia

Cada servicio tiene su propia base de datos aislada. Pedidos no puede consultar directamente la tabla de usuarios — debe llamar al servicio de Usuarios por HTTP. Esto garantiza el desacoplamiento real entre servicios.

Código — main.py

# usuarios/main.py
from fastapi import FastAPI, HTTPException, Depends
from pydantic import BaseModel, EmailStr
from sqlalchemy.orm import Session
from .database import engine, get_db
from . import models

# Crea la tabla si no existe al arrancar
models.Base.metadata.create_all(bind=engine)

app = FastAPI(title="Servicio de Usuarios", version="1.0.0")

@app.post("/usuarios/", status_code=201)
def crear_usuario(usuario: UsuarioCreate, db: Session = Depends(get_db)):
    existe = db.query(models.Usuario).filter(
        models.Usuario.email == usuario.email
    ).first()
    if existe:
        raise HTTPException(409, "Email ya registrado")
    nuevo = models.Usuario(**usuario.dict())
    db.add(nuevo)
    db.commit()
    db.refresh(nuevo)
    return nuevousuarios/main.py
Conceptos FastAPI en este archivo

Depends(get_db) — inyección de dependencias: FastAPI abre la sesión de base de datos automáticamente y la cierra al terminar la petición.

HTTPException(409) — respuesta de error con código de estado HTTP semánticamente correcto: 409 Conflict significa "el recurso ya existe".

response_model — FastAPI valida y filtra la respuesta usando el schema Pydantic indicado, evitando exponer campos internos como la contraseña.

Explicación detallada — db: Session = Depends(get_db)

Esta línea aparece en todos los endpoints que acceden a la base de datos. Tiene cuatro partes:

ParteQué esPara qué sirve
db Nombre de la variable Dentro del endpoint se usa como db.query(), db.add(), db.commit()
: Session Anotación de tipo Le dice a Python y a PyCharm que db es una sesión SQLAlchemy — activa el autocompletado
= Depends(...) Mecanismo de FastAPI FastAPI lo reconoce y ejecuta la función indicada antes de llamar al endpoint
get_db Función generadora Abre la sesión, la entrega con yield, y la cierra en el finally

¿Por qué no abrir la sesión directamente en el endpoint?

La forma ingenua tendría un problema grave:

# ❌ Forma problemática — la sesión puede quedar abierta
def crear_usuario(usuario: UsuarioCreate):
    db = SessionLocal()        # abro la sesión
    nuevo = models.Usuario(...)
    db.add(nuevo)
    db.commit()
    db.close()                 # si hay un error antes, nunca se ejecuta
    return nuevo
# ✅ Forma correcta — get_db() garantiza que siempre se cierra
def get_db():
    db = SessionLocal()   # 1. abre la sesión
    try:
        yield db          # 2. la entrega al endpoint
    finally:
        db.close()        # 3. la cierra SIEMPRE, haya error o no

Flujo completo paso a paso

Cliente envía POST /usuarios/

FastAPI recibe la petición y analiza los parámetros del endpoint.

FastAPI detecta Depends(get_db)

Ejecuta get_db(), que abre la sesión y hace yield db.

FastAPI llama al endpoint con db inyectada

El código del endpoint usa db.query(), db.add(), db.commit() normalmente.

El endpoint termina (con éxito o con error)

FastAPI ejecuta el finally de get_db() → db.close().

FastAPI responde al cliente

La sesión ya está cerrada. El cliente recibe el resultado.

¿Por qué se llama "inyección de dependencias"?

Porque el endpoint no crea ni destruye la sesión — alguien se la inyecta desde afuera. El endpoint solo declara que la necesita (db: Session = Depends(get_db)) y FastAPI se encarga del resto. El cliente que llama al endpoint no sabe que existe db — es completamente invisible para él.

Las tres clases del servicio de Usuarios

El servicio define dos clases en main.py y una clase en models.py. Cada una tiene una responsabilidad distinta y ninguna puede reemplazar a las otras.

ClaseArchivoHereda deResponsabilidad
Usuario models.py Base (SQLAlchemy) Define la tabla en la base de datos — columnas, tipos SQL, índices
UsuarioCreate main.py BaseModel (Pydantic) Valida los datos que envía el cliente al crear un usuario
UsuarioPublico main.py BaseModel (Pydantic) Define qué campos devuelve la API — excluye la contraseña

Usuario — models.py

Representa la tabla en la base de datos. SQLAlchemy la usa para crear la tabla y para leer y escribir registros. Tiene los 5 campos reales que se guardan en disco, incluyendo password. 

class Usuario(Base):           # hereda de Base → SQLAlchemy lo gestiona
    __tablename__ = "usuarios"  # nombre de la tabla en SQLite

    id        = Column(Integer, primary_key=True)  # autogenerado
    nombre    = Column(String,  nullable=False)
    email     = Column(String,  unique=True)
    password  = Column(String,  nullable=False)    # se guarda en BD
    creado_en = Column(DateTime, default=datetime.utcnow)  # autogeneradomodels.py

UsuarioCreate — main.py

Representa los datos que el cliente envía al hacer POST /usuarios/. Solo tiene los 3 campos que el cliente debe proporcionar — sin id (lo genera la BD) y sin creado_en (lo genera el servidor). 

class UsuarioCreate(BaseModel):  # hereda de BaseModel → Pydantic valida
    nombre:   str
    email:    EmailStr             # Pydantic valida que sea un email válido
    password: strmain.py

UsuarioPublico — main.py

Representa los datos que la API devuelve. Deliberadamente excluye password. Aunque el objeto Usuario traiga la contraseña de la BD, FastAPI nunca la incluye en la respuesta porque UsuarioPublico no la declara. 

class UsuarioPublico(BaseModel):  # solo 4 campos — password ausente
    id:        int
    nombre:    str
    email:     str
    creado_en: datetime

    class Config:
        from_attributes = True  # permite leer desde objetos SQLAlchemymain.py

El flujo de las tres clases juntas

Cliente envía el body

{"nombre": "Ana", "email": "ana@ejemplo.com", "password": "segura123"}

Pydantic valida con UsuarioCreate

Comprueba que los tres campos existen y que el email tiene formato válido. Si falla, devuelve 422 automáticamente.

Se crea un objeto Usuario (models.py) y se guarda en BD

SQLAlchemy asigna el id y el creado_en automáticamente.

FastAPI filtra la respuesta con UsuarioPublico

Devuelve solo id, nombre, email y creado_en. La contraseña nunca sale.

¿Por qué no usar una sola clase para todo?

Una sola clase no puede cubrir los tres propósitos simultáneamente. models.py necesita columnas SQL que Pydantic no entiende. UsuarioCreate no debe tener id ni creado_en porque el cliente no los envía. UsuarioPublico debe excluir password para no exponerla. Cada clase tiene exactamente los campos que su capa necesita — ni más, ni menos.

06 — Servicio

Servicio de Pedidos

El servicio más complejo: orquesta una transacción que involucra a Usuarios y Productos. Demuestra la comunicación real entre microservicios.

Flujo interno al crear un pedido

1 — Recibe la petición

El cliente envía POST /pedidos/ con usuario_id e items.

2 — Verifica el usuario

Llama a GET http://localhost:8001/usuarios/{id}. Si responde 404, aborta con error.

3 — Verifica stock de cada producto

Por cada item, llama a GET http://localhost:8002/productos/{id} y comprueba que hay suficiente stock.

4 — Descuenta el stock

Llama a PATCH /productos/{id}/stock con cantidad negativa para cada producto.

5 — Guarda y responde

Guarda el pedido en pedidos.db y devuelve 201 Created con el detalle completo.

# Fragmento clave — pedidos/main.py
@app.post("/pedidos/", status_code=201)
async def crear_pedido(pedido: PedidoCreate, db: Session = Depends(get_db)):

    async with httpx.AsyncClient() as client:

        # Llamada al servicio de Usuarios
        resp = await client.get(f"{USUARIOS_URL}/usuarios/{pedido.usuario_id}")
        if resp.status_code == 404:
            raise HTTPException(404, "Usuario no encontrado")

        # Llamada al servicio de Productos (por cada item)
        for item in pedido.items:
            resp = await client.get(f"{PRODUCTOS_URL}/productos/{item.producto_id}")
            producto = resp.json()
            if producto["stock"] < item.cantidad:
                raise HTTPException(409, "Stock insuficiente")pedidos/main.py
async / await — ¿por qué aquí?

Las llamadas HTTP a otros servicios son operaciones de entrada/salida que toman tiempo. Con async/await, FastAPI puede atender otras peticiones mientras espera la respuesta del servicio de Usuarios o Productos, en lugar de bloquear el servidor.

¿Por qué Pedidos tiene schemas.py y los demás no?

La decisión es de legibilidad. Cuando un servicio tiene pocos schemas simples, conviven bien dentro de main.py. Cuando los schemas son muchos o complejos, separarlos en schemas.py mantiene cada archivo enfocado en una sola responsabilidad.

ServicioSchemasDónde vivenMotivo
Usuarios UsuarioCreate, UsuarioPublico main.py 2 clases cortas, no saturan el archivo
Productos ProductoCreate, ProductoPublico, StockUpdate main.py 3 clases simples, manejables en el mismo archivo
Pedidos PedidoCreate, PedidoPublico, ItemPedido, ItemDetalle, EstadoPedido schemas.py 5 clases con anidamiento — se separan para no saturar main.py

¿Qué hace complejo el schema de Pedidos?

Un pedido contiene una lista de items, y cada item es a su vez un objeto con sus propios campos. Eso crea schemas anidados: 

# Un pedido contiene una lista de ItemPedido
class ItemPedido(BaseModel):
    producto_id: int
    cantidad:    int

class PedidoCreate(BaseModel):
    usuario_id: int
    items:      list[ItemPedido]  # ← schema dentro de schema

# La respuesta también tiene sus propios items con más detalle
class ItemDetalle(BaseModel):
    producto_id: int
    nombre:      str
    precio:      float
    cantidad:    int
    subtotal:    float

class PedidoPublico(BaseModel):
    id:         int
    usuario_id: int
    items:      list[ItemDetalle]  # ← schema dentro de schema
    total:      float
    estado:     EstadoPedido
    creado_en:  strpedidos/schemas.py

Si estas 5 clases estuvieran en main.py, el archivo mezclaría schemas, lógica de negocio, llamadas HTTP a otros servicios y configuración de la app. Separar los schemas en su propio archivo deja cada archivo con una sola responsabilidad clara.

Convención en proyectos reales

En proyectos profesionales todos los servicios suelen tener schemas.py desde el principio, anticipando que crecerán. Aquí Usuarios y Productos se dejaron con los schemas en main.py intencionalmente, para que puedas comparar ambas formas y entender cuándo y por qué se hace la separación.

07 — Servicio

API Gateway

Punto de entrada único para el cliente. Recibe todas las peticiones, añade un ID de correlación para trazabilidad y las reenvía al servicio correcto.

¿Cómo funciona el enrutamiento?

El Gateway usa una ruta genérica que captura el primer segmento de la URL como nombre del servicio y el resto como la ruta a reenviar: 

# El cliente llama a:
POST http://localhost:8000/usuarios/usuarios/

# El Gateway extrae:
service = "usuarios"   # primer segmento
path    = "usuarios/"  # resto de la ruta

# Y reenvía a:
POST http://localhost:8001/usuarios/  # SERVICES[service] + pathgateway/main.py

Trazabilidad con Correlation ID

Cada petición recibe un identificador único (X-Correlation-ID) que viaja en los headers. En un sistema real, este ID permite rastrear una petición a través de todos los servicios que involucra en los logs. 

correlation_id = str(uuid.uuid4())  # ej: "a3f2-8b1c-..."

# Se añade al reenvío
headers={"X-Correlation-ID": correlation_id}

# Y a la respuesta al cliente
response.headers["X-Correlation-ID"] = correlation_id
response.headers["X-Response-Time-Ms"] = str(elapsed)gateway/main.py
08 — Componente

database.py — Conexión a la base de datos

Cada servicio tiene su propio database.py que configura la conexión a su base de datos SQLite independiente.

SQLite es una base de datos que vive en un único archivo en disco. No requiere instalar ningún servidor — es perfecta para desarrollo y aprendizaje. En producción se reemplazaría por PostgreSQL o MySQL sin cambiar el resto del código. 

# usuarios/database.py (igual estructura en productos/ y pedidos/)
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, declarative_base

# Cada servicio apunta a su propio archivo .db
SQLALCHEMY_DATABASE_URL = "sqlite:///./usuarios.db"

# Engine: objeto que gestiona la conexión a la BD
engine = create_engine(
    SQLALCHEMY_DATABASE_URL,
    connect_args={"check_same_thread": False}  # necesario para SQLite
)

# Fábrica de sesiones: cada petición obtiene su propia sesión
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

# Base para los modelos ORM
Base = declarative_base()

# Generador de sesiones — usado con Depends() en FastAPI
def get_db():
    db = SessionLocal()
    try:
        yield db       # entrega la sesión al endpoint
    finally:
        db.close()     # la cierra siempre, aunque haya errordatabase.py
¿Qué es yield en get_db()?

get_db() es un generador. FastAPI lo usa con Depends(): ejecuta el código hasta el yield (abre la sesión), entrega la sesión al endpoint, y cuando el endpoint termina ejecuta el finally (cierra la sesión). Esto garantiza que ninguna sesión quede abierta aunque ocurra un error.

09 — Componente

models.py — Tablas de la base de datos

Define las tablas de la base de datos como clases Python usando el ORM de SQLAlchemy. Cada clase es una tabla; cada atributo de clase, una columna.

ORM significa Object-Relational Mapper: permite trabajar con la base de datos usando objetos Python en lugar de escribir SQL directamente. 

# usuarios/models.py
from sqlalchemy import Column, Integer, String, DateTime
from .database import Base

class Usuario(Base):
    __tablename__ = "usuarios"  # nombre de la tabla en SQLite

    id        = Column(Integer, primary_key=True, index=True)
    nombre    = Column(String,  nullable=False)
    email     = Column(String,  unique=True, index=True, nullable=False)
    password  = Column(String,  nullable=False)
    creado_en = Column(DateTime, default=datetime.utcnow)usuarios/models.py
# productos/models.py
class Producto(Base):
    __tablename__ = "productos"

    id          = Column(Integer, primary_key=True, index=True)
    nombre      = Column(String,  nullable=False)
    precio      = Column(Float,   nullable=False)
    stock       = Column(Integer, default=0)
    descripcion = Column(String,  nullable=True)productos/models.py
# pedidos/models.py
class Pedido(Base):
    __tablename__ = "pedidos"

    id         = Column(Integer, primary_key=True, index=True)
    usuario_id = Column(Integer, nullable=False)

models.py vs schemas.py — diferencia fundamental

models.py define cómo se guardan los datos en la base de datos (columnas, tipos SQL, índices). schemas.py define cómo se ven los datos en la API (validación de entrada, formato de respuesta). Son dos capas separadas con propósitos distintos.

10 — Componente

schemas.py — Modelos Pydantic

Define los contratos de la API: qué datos acepta cada endpoint y qué devuelve. Pydantic valida automáticamente que los datos tengan el tipo y formato correcto.

Solo el servicio de Pedidos tiene un schemas.py separado porque sus modelos son más complejos: incluye listas de items, totales calculados y estados. En Usuarios y Productos los schemas están directamente en main.py. 

# pedidos/schemas.py
from pydantic import BaseModel, Field
from enum import Enum

# Schema de ENTRADA — lo que envía el cliente
class ItemPedido(BaseModel):
    producto_id: int
    cantidad:    int = Field(..., gt=0)  # gt=0 significa mayor que cero

class PedidoCreate(BaseModel):
    usuario_id: int
    items:      list[ItemPedido]

# Schema de SALIDA — lo que devuelve la API
class PedidoPublico(BaseModel):
    id:         int
    usuario_id: int
    items:      list[ItemDetalle]
    total:      float
    estado:     EstadoPedido
    creado_en:  strpedidos/schemas.py
Validación automática con Pydantic

Si un cliente envía "cantidad": -5 o "cantidad": "hola", FastAPI rechaza la petición automáticamente con un 422 Unprocessable Entity antes de que el código del endpoint se ejecute. No necesitas escribir validaciones manuales.

12 — Script

reset.py — Limpieza de datos

Elimina todos los datos de los tres servicios llamando a sus endpoints de reset, dejando el sistema listo para ejecutar demo.py nuevamente.

El orden importa: primero se borran los Pedidos (que referencian usuarios y productos), luego los Usuarios, y por último los Productos se restablecen a los datos iniciales (en lugar de quedar vacíos). 

# reset.py
servicios = [
    ("Pedidos",   f"{GATEWAY}/pedidos/pedidos/reset"),
    ("Usuarios",  f"{GATEWAY}/usuarios/usuarios/reset"),
    ("Productos", f"{GATEWAY}/productos/productos/reset"),
]

for nombre, url in servicios:
    resp = httpx.delete(url)
    print(f"{nombre} → {resp.json()['mensaje']}")reset.py
Solo para desarrollo

14 — Concepto

Decoradores — @app.get, @app.post

Un decorador es una función que envuelve a otra función y le añade comportamiento extra sin modificar su código interno.

En Python, un decorador se escribe con @ justo encima de una función. FastAPI usa decoradores para registrar qué URL y qué método HTTP corresponde a cada función del código.

¿Qué hace el decorador exactamente?

@app.get("/usuarios/{usuario_id}", response_model=UsuarioPublico)
def obtener_usuario(usuario_id: int, db: Session = Depends(get_db)):
    ...

Ese decorador le dice a FastAPI tres cosas:

Parte del decoradorQué indica
@app.getResponde a peticiones con método HTTP GET
"/usuarios/{usuario_id}"La URL que activa esta función. {usuario_id} es un parámetro variable
response_model=UsuarioPublicoFiltra la respuesta usando ese schema Pydantic

Decoradores disponibles en FastAPI

DecoradorMétodo HTTPUso típico en el proyecto
@app.getGETListar usuarios, obtener producto por ID
@app.postPOSTCrear usuario, crear pedido
@app.patchPATCHActualizar stock de un producto
@app.deleteDELETEResetear base de datos
@app.api_routeVariosGateway — acepta todos los métodos a la vez
Sin el decorador, FastAPI no sabe que la función existe

Una función sin @app.get o similar es simplemente una función Python normal — no es un endpoint. FastAPI solo registra las funciones que tienen un decorador de ruta. Por eso get_db() y seed() no tienen decorador: son funciones auxiliares, no endpoints.

15 — Concepto

def vs async def

La diferencia está en cómo el servidor maneja la espera. Un endpoint con async def puede ceder el control mientras espera, permitiendo atender otras peticiones en ese tiempo.

¿Cuándo usar cada uno?

def normalasync def
Usado enUsuarios, ProductosPedidos, Gateway
Cuándo usarloSolo accede a la base de datos localHace llamadas HTTP a otros servicios
Por quéSQLAlchemy gestiona su propia esperahttpx requiere async/await para no bloquear

Ejemplo comparado

# def normal — solo BD local, no necesita async
@app.get("/usuarios/{usuario_id}")
def obtener_usuario(usuario_id: int, db: Session = Depends(get_db)):
    return db.query(models.Usuario).filter(...).first()
    # SQLAlchemy habla con SQLite — no necesita await
# async def — llama a otros servicios por HTTP
@app.post("/pedidos/")
async def crear_pedido(pedido: PedidoCreate, db: Session = Depends(get_db)):
    async with httpx.AsyncClient() as client:
        resp = await client.get(...)   # espera la respuesta de Usuarios
        resp = await client.get(...)   # espera la respuesta de Productos
        # mientras espera, el servidor puede atender otras peticiones
Analogía

def normal es como un cajero que atiende a una persona a la vez y no puede hacer nada más mientras espera. async def es como un cajero que, mientras espera que el cliente busque su tarjeta, atiende brevemente a otra persona — luego vuelve cuando el primero está listo.

16 — Concepto

httpx — el cliente HTTP

httpx es la librería que usan los servicios para hacerse llamadas entre sí. Es como un navegador web dentro del código Python.

Cuando el servicio de Pedidos necesita verificar que un usuario existe, no puede llamar directamente a una función de Usuarios — son aplicaciones separadas. La única forma es hacer una petición HTTP, igual que lo haría un navegador. httpx es la herramienta que hace eso desde Python.

Uso en el proyecto

import httpx

# Crear un cliente asíncrono (se cierra automáticamente al salir del bloque)
async with httpx.AsyncClient() as client:

    # GET — obtener datos
    resp = await client.get("http://localhost:8001/usuarios/1")

    # POST — enviar datos
    resp = await client.post("http://localhost:8002/productos/",
                              json={"nombre": "Mouse", "precio": 29.99})

    # PATCH — modificar parcialmente
    resp = await client.patch("http://localhost:8002/productos/1/stock",
                               json={"cantidad": -1})

# Leer el resultado
print(resp.status_code)   # 200, 201, 404, etc.
print(resp.json())        # diccionario Python con los datos

ORM y métodos de SQLAlchemy

ORM significa Object-Relational Mapper. Permite trabajar con la base de datos usando objetos Python en lugar de escribir SQL directamente.

Sin ORM vs con ORM

# Sin ORM — SQL directo
cursor.execute("INSERT INTO usuarios (nombre, email) VALUES (?, ?)",
               (usuario.nombre, usuario.email))
conexion.commit()

# Con ORM — objetos Python
nuevo = models.Usuario(nombre=usuario.nombre, email=usuario.email)
db.add(nuevo)
db.commit()

Métodos usados en el proyecto

db.query(Modelo)SELECT * FROM tablaInicia una consulta sobre una tabla

Ejemplo completo — crear un usuario

# 1. Consultar si el email ya existe
existe = db.query(models.Usuario)           # SELECT * FROM usuarios
         .filter(models.Usuario.email ==    # WHERE email =
                 usuario.email).first()     # LIMIT 1

# 2. Crear el objeto Python
nuevo = models.Usuario(
    nombre=usuario.nombre,
    email=usuario.email,
    password=usuario.password,
)

# 4. Guardar en disco

18 — Concepto

Códigos de estado HTTP en el proyecto

Cada respuesta HTTP incluye un código numérico que indica si la operación tuvo éxito o qué tipo de error ocurrió.

CódigoNombreCuándo ocurre en el proyecto
200 OK Éxito GET exitoso — obtener usuario, listar productos
201 Created Creado POST exitoso — usuario creado, pedido confirmado
404 Not Found No encontrado Usuario o producto con ese ID no existe en la BD
409 Conflict Conflicto Email ya registrado, o stock insuficiente para el pedido
422 Unprocessable Datos inválidos Pydantic rechaza el body — falta un campo, tipo incorrecto, email mal formado
500 Internal Error Error del servidor Error inesperado en el código — bug no controlado

Cómo se usan en el código

# 201 — se indica en el decorador
@app.post("/usuarios/", status_code=201)
def crear_usuario(...):
    ...

# 404 — se lanza manualmente con HTTPException
if not usuario:
    raise HTTPException(404, "Usuario no encontrado")

# 409 — conflicto de negocio
if producto["stock"] < item.cantidad:
    raise HTTPException(409, "Stock insuficiente")

# 422 — lo genera Pydantic automáticamente, no hay que escribirlo
200 vs 201

Ambos indican éxito, pero 201 es más específico: significa que se creó un recurso nuevo. Por convención, los endpoints POST que crean registros devuelven 201, y los GET devuelven 200.

24 — Referencia

Preguntas frecuentes

¿Por qué SQLite y no PostgreSQL?

SQLite no requiere instalar ni configurar ningún servidor — el archivo de base de datos se crea automáticamente al arrancar el servicio. Para aprender los conceptos de microservicios, SQLite es suficiente. El código de SQLAlchemy es idéntico para ambos; solo cambiaría la URL de conexión en database.py.

¿Por qué necesito 4 terminales?

Porque cada microservicio es un proceso independiente que debe estar corriendo simultáneamente. En un entorno real correrían en servidores o contenedores Docker separados. Las 4 terminales simulan ese escenario en una sola máquina.

¿Qué pasa si apago el servicio de Usuarios mientras Pedidos está corriendo?

Si alguien intenta crear un pedido, Pedidos intentará llamar a Usuarios y recibirá un error de conexión (ConnectionRefusedError). El pedido fallará con un 500. Los datos existentes en las demás bases de datos no se afectan — solo fallan las operaciones nuevas que requieren ese servicio.

¿Los datos se pierden si reinicio un servicio?

No. Los datos están en archivos SQLite (usuarios.db, productos.db, pedidos.db). Al reiniciar el servicio, SQLAlchemy encuentra los archivos existentes y los usa. Solo se pierden si se borran los archivos .db o se ejecuta python reset.py.

¿Por qué el Swagger del Gateway no tiene formularios?

El Gateway usa una ruta genérica /{service}/{path:path} que acepta cualquier URL. Swagger no puede generar formularios para una ruta que no tiene schema definido. Para probar con formularios, usar el Swagger de cada servicio directamente (:8001/docs, :8002/docs, :8003/docs).

¿Por qué el servicio de Pedidos no tiene models.py en la presentación original?

La presentación mostraba una versión simplificada donde Pedidos usaba un diccionario en memoria. En este proyecto se añadió models.py y database.py para que Pedidos sea consistente con los demás servicios y los pedidos persistan al reiniciar.

¿Puedo agregar más servicios?

Sí. Basta con crear una nueva carpeta con __init__.py, main.py, models.py y database.py, arrancarla en un nuevo puerto, y agregar su URL al diccionario SERVICES en gateway/main.py. Los demás servicios no necesitan ningún cambio.
